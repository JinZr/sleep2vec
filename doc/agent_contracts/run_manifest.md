# Run Manifest Contract

The experiment workspace `run_manifest.tsv` is the human entry point for managed runs. Each row records `experiment_id`, `step_id`, stable `run_id`, semantic `run_name`, runtime `version`, parameter summary, frozen config/script paths, runtime artifact links, status, and lifecycle timestamps.

Stable ids use `run-000`, `run-001`, and so on within each step, and continue across later plans for that step. They are never the sole description of a run. Every run also receives a semantic name, such as `baseline` or `lr-2e-6__router-frozen__class-weights-off`; generated runtime versions include experiment id, step id, stable run id, and that semantic name.

The canonical managed identity is `(step_id, run_id)`. Both fields are required in every current managed row, and `run_name` never participates in matching. Evidence with a complete managed identity either matches that exact key or remains unmatched; it never falls back to another run by `version`. Runtime `version` is an external evidence key only when managed identity is unavailable, and version-only evidence may update a managed row only when it matches exactly one run.

Once a managed row exists, its experiment/run identity, semantic name, parameter summary, runtime version, config and script snapshots and hashes, managed directories, artifact link, and `runtime.*` / `yaml:/...` parameter values are frozen. Management locators written by the planner, workspace, or adaptive workflow are absolute local paths: this includes plan/run/report directories and frozen config, script, artifacts, runtime, and checkpoint locations. Recipe search keys keep that same spelling in `plan.json`, `run.json`, TSV manifests, rankings, and adaptive artifacts. Later observations may omit frozen fields but cannot erase or change them. User-authored semantic dataset and checkpoint inputs retain their declared path semantics and are not normalized merely because they appear in a recipe.

Historical plans and tables that use the removed identity format are read-only. Current launch, monitor, stop, selection, adaptive, and experiment mutation commands reject them before starting a process, terminating a PID, or writing a file. There is no migration, alias, or automatic aggregation path.

Canonical reads have one lifecycle contract. Only fresh experiment initialization may observe a confirmed missing `run_manifest.tsv`; initialization creates the canonical `step_id`/`run_id` header before any run is registered. Thereafter, a valid header-only table means that the experiment currently has no runs, while a zero-byte, malformed, unreadable, wrong-header, legacy, duplicate-key, recursive-alias, or dangling-symlink artifact is corrupt and fails before mutation. Authoritative experiment and step YAML reject duplicate keys and recursive aliases rather than accepting ambiguous ownership or unbounded graphs. Managed auxiliary tables use strict identity headers before row validation; when a managed producer has zero rows it still writes `step_id` and `run_id`, so a successful empty artifact remains readable. Local and SSH callers use the workspace-owned reader, so an SSH timeout, permission error, target-type error, or non-missing return code is never interpreted as absence. Optional non-managed evidence may remain best-effort, but no caller may read or write the canonical manifest directly.

Each managed run directory contains:

- `run.json`: identity, parameters, status, and frozen snapshot paths.
- `config.yaml`: the exact generated config used by the run.
- `launch.sh`: the exact launch command.
- `artifacts.json`: links to heavyweight runtime outputs such as `log-finetune/`, checkpoints, W&B, or datasets.

Experiment checkpoint indexing follows each managed row's frozen `checkpoint_dir` and reads its run manifest from the frozen `runtime_dir`. The two paths must be present together; a row with both paths empty is a valid non-checkpoint-producing run. Before replacing or ranking from the checkpoint manifest, every existing checkpoint row must still belong to an eligible managed run, and every non-empty `checkpoint_path` must be a direct child of that run's frozen `checkpoint_dir`. Empty checkpoint evidence remains valid. It does not recursively scan the experiment root or infer identity from directory names. Checkpoints remain in their canonical runtime location and are not copied into the workspace.

Config and script hashes are recorded in the plan, per-run JSON, and canonical manifest before launch. Every hparam operation verifies the actual frozen files through the shared plan preflight before it starts or stops a process or writes an artifact. The generated script's enforced working directory is also the source of its frozen `runtime_dir` and `checkpoint_dir`. Lifecycle changes are appended to the experiment-level `events.jsonl`; stops require a reason, and `run_launched` means that the current command invocation actually started the process successfully.

Every hparam writer validates that `plan.json` is contained by the declared experiment root, is registered in the initialized step manifest, matches the experiment and step metadata, gives every run the same experiment/step binding, and has a complete matching row in the canonical run manifest. Missing or partial canonical state fails instead of being repaired by selection or postprocessing. Threshold and ensemble analysis therefore require a current managed plan and managed candidate identities. A copied or structurally plausible standalone plan cannot launch or write analysis artifacts outside its registered managed location. Launch also rejects a completed experiment before creating directories, starting processes, or writing state, and merges canonical workspace status with local launch/status evidence before deciding eligibility.

W&B synchronization joins an existing managed row only with external ownership proof. Evidence with a non-empty `experiment_id` must match that experiment before its managed identity can be considered; evidence without `experiment_id` ignores any unproven step/run claim and may resolve only by one unique runtime `version`. W&B identity and display name are resolution-only; canonical rows preserve the frozen managed version and accept only W&B status/state, W&B identifiers/URLs/group metadata, and W&B timestamps. Mutation-facing W&B tracking produces managed-keyed observations only; it never writes a complete canonical snapshot. Managed metric rows receive their identity and version from the matched canonical run. Unmatched or foreign W&B runs remain only in the raw W&B inventory and history and do not enter canonical run, metric, or ranking tables. A W&B `running` state remains non-terminal when PID evidence is unavailable.

Workspace-scoped launch, status, metric, checkpoint, ranking, adaptive-registry, and candidate evidence proves scope through an already validated workspace location plus managed key. Candidate and ranking tables, including an existing checkpoint-level ranking that would be replaced, are matched against the complete workspace canonical manifest before legal other-step filtering or runtime scanning; retained candidates additionally belong to the current frozen plan. Rank/top-k filtering must leave at least one candidate before a runnable postprocess producer creates directories or files. Before evidence is allowlisted or overlaid with plan fields, every supplied non-empty experiment id, version, config, snapshot hash, parameter, or frozen management locator must agree with its canonical or plan row; unmatched, foreign, or drifting evidence fails rather than being silently discarded. Postprocessing resolves its local plan directory once at entry, and generated replay scripts freeze absolute plan and candidate-table locators plus the repository cwd/PYTHONPATH bootstrap, so generated configs, logits, manifests, summaries, scripts, and returned management paths remain stable across cwd changes.

Run-row status merging has one precedence rule. `run_manifest.tsv` is the durable status owner, `run_status.tsv` is its plan-local mirror, `launch_manifest.tsv` supplies execution evidence, and runtime `run_manifest.json` supplies metrics and checkpoint evidence only. Existing terminal states remain sticky, except that incoming `failed` evidence may correct `completed` or `finished`. Non-terminal rows accept incoming observations except that an active `launched`, `running`, `unknown_remote`, or `missing_pid` state cannot be regressed by stale `planned` or `pending` mirror evidence; an update without status preserves the existing value. The canonical manifest owner re-reads and commits the table for both local and SSH workspaces, verifies every newly introduced key against the workspace experiment id, returns the complete rows actually written, and passes those rows directly to the run-matrix writer. Launch, monitor, stop, adaptive supersede, W&B sync, and experiment monitoring use the returned rows for mirrors, reports, and transition events; no caller writes `run_manifest.tsv` directly. Missing launch evidence never synthesizes a planned state. PID reads use the same explicit-missing rule as canonical reads: only a confirmed absent file returns no PID. Empty, non-numeric, non-positive, or invalid-encoding local PID content is confirmed corrupt, so monitoring maps launchable `planned` or `pending` state to non-launchable `missing_pid`. A local path/read `OSError` while a run is still planned or pending aborts before canonical or local mutation, allowing a later confirmed absence to remain launchable; other states retain normal reducer precedence. Remote permission, type, decoding, transport, and timeout failures produce recoverable `unknown_remote` evidence. Stop propagates PID-read uncertainty before kill or mutation. Stopping an already-terminal canonical run fails before PID access or mutation. Status vocabulary and the `finished` to `completed` monitor normalization remain unchanged.

Commands that treat `run_manifest.tsv` as authoritative must distinguish an empty valid table from an unavailable one. In particular, `collect-runs` requires an explicit managed workspace root, fails when the canonical manifest is missing, unreadable, or invalid, and rejects any output path that resolves to or shares an inode with the canonical table; only optional W&B summaries remain best-effort evidence.

Every managed table declares its identity cardinality when validated. Run, launch, status, ranking, and adaptive registry tables allow one row per `(step_id, run_id)`; metric, checkpoint, checkpoint-ranking, and selected-checkpoint evidence tables allow multiple rows per managed run. Both forms still require a complete managed identity and reject removed-format columns.

Training and preset runtime sidecar manifests remain JSON files with `kind`, timestamps, input paths, output paths, and relevant metrics or counts.

Finetune run manifests live at `log-finetune/<version>/run_manifest.json`.
Preset sidecar manifests live next to generated preset pickle files as `<preset>.manifest.json`.
