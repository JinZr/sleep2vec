## Purpose

Generate agent context bundles, validate task recipes, enforce stop-and-consult policy, and create safe command plans around the existing canonical runtime entrypoints.

## Layers

- `skills/`: human-readable task playbooks and examples.
- `recipes/`: declarative task cards for finetune, preset, inference, tuning, and sleep2stat plans.
- `agent_policies/`: consultation policy and approved defaults.
- `agent_tools/`: lightweight Python package for summaries, decision gates, context bundles, plans, and run collection.
- `doc/agent_contracts/`: context bundle, recipe, run manifest, user decision, and external-test-locking contracts.

Inside `agent_tools`, `decisions.py`, `plans.py`, `hparam.py`, and `experiments.py` are stable public entrypoints. Their implementation owners are responsibility-based: `decision_*` for gate types/paths/rules, `plan_*` for rendering/context/hparam compilation, `hparam_*` plus `run_artifacts`/`run_evidence` for run execution and analysis, and `experiment_io`/`experiment_tracking` for transport and experiment evidence. Leaf modules must not import those facades or `cli.py`.

## Stop-And-Consult Contract

Agent tooling returns:

- `PASS`: safe to continue.
- `WARN`: safe to continue with warnings.
- `NEEDS_USER_INPUT`: exit code `2`; ask the generated questions and do not generate runnable training scripts.
- `FAIL`: exit code `1`; invalid or missing required input.

High-impact decisions must come from explicit user decisions, explicit CLI arguments, explicit recipe fields, or explicit config fields. Approved defaults are reserved for low-impact runtime convenience values.

## Context Bundles

`agent_tools doctor` is read-only unless `--output-dir` is explicitly supplied. When an output directory is supplied and blocking issues exist, it writes diagnostic `questions.json` and `questions.md` without applying experiment overwrite policy.

`agent_tools context` is diagnostic-only. It writes `context.json`, `context.md`, `questions.json`, `questions.md`, and `commands.blocked.sh`, while runnable commands require a managed recipe-backed `agent_tools plan`; `consultation_required` is true when any issue needs user input, even if another issue makes the overall status `FAIL`.

Context bundles include skill metadata, owners, relevant index docs, expected agent artifacts, and best-effort index or preset summaries when the config points to `data.finetune_data_index` or `data.finetune_preset_path`. For survival and multilabel configs, index summaries include subject key coverage and sidecar-key checks. For `task=sleep2stat`, NPZ index summaries use `sleep2stat.data.index`.

## Plan Generation

`agent_tools plan` canonicalizes a local output directory at the public entry, then uses the shared read-only preflight before writing scripts. New runnable plans require resolved `experiment` and `step` metadata, and hparam recipes must declare that outer ownership locally rather than inheriting it from their base finetune recipe. A user-selected effective config is loaded and passed through config summary plus consultation before workspace mutation; hparam selection updates the base-recipe owner used for run snapshots, while explicit empty and `ASK_USER` config decisions remain unresolved instead of falling back to the recipe config. The output directory must be inside `experiment.root`. If ownership metadata or the selected config is unresolved or invalid, the CLI returns diagnostics without creating directories or files. If the workspace identity is resolved but another decision blocks, the command initializes the managed experiment/step and writes `plan.blocked.md` plus questions without registering a run; a corrected recipe can retry in a new plan directory under the same root. Passing plans freeze the resolved recipe/config/command snapshots and register planned runs. Runnable non-hparam finetune, infer, evaluate, preset-prepare, and sleep2stat scripts use `REPO_ROOT` for cwd/PYTHONPATH, commit `running` before runtime and `completed/failed` on exit through the canonical owner, and reject terminal replay. Step recipe/plan entries, plan/run/config/script/artifacts locations, derived runtime/checkpoint directories, and matching event locators are absolute local management paths, so later commands do not depend on their cwd. User-authored semantic input paths retain their recipe semantics. Hyper-parameter scripts use validation-only selection and `--no-test-after-fit` by default; they may run test-after-fit only when `external_test_locked=false`, `final_test_unlocked=true`, and `test_after_fit=true` are explicit. Final external-test scripts require explicit final-test unlock plus an explicit existing checkpoint path, and YAML-overridden hparam searches also require an explicit selected final-eval config path.

Recipe `variant` controls the generated module namespace for model tasks. Finetune, hparam, infer, and evaluate accept `sleep2vec`, `sleep2vec2`, `sleep2expert`, and `sex_age_baseline`; preset preparation accepts the first three because the age/sex baseline has no preset-generation surface. `task=sleep2stat` is variantless. Missing, extra, or unsupported variants block command generation instead of falling back to another namespace. The finite matrix is pinned in `tests/agent_tools/test_agent_tools_cli_contract.py`.

Sleep2stat plans call only existing `sleep2stat` CLI commands: `python -m sleep2stat validate-config`, optional YASA record preflight through `python -m sleep2stat validate-config --check-records`, `python -m sleep2stat run`, optional `python -m sleep2stat summarize`, and optional `python -m sleep2stat plot-cohort`. Summarize and cohort plotting always use config `run.output_dir`; a recipe `artifacts.run_dir` that differs from config `run.output_dir` blocks before scripts are written. Agent checks rely on `sleep2stat.config.load_config()` for structural/runtime semantics and only add agent-facing risk gates such as placeholder model analyzer configs or checkpoints. Agent tooling must not translate stale sleep2stat YAML fields or reducer aliases; current `sleep2stat.config` is the schema source.

Recipe `base_recipe` paths are resolved relative to the local recipe file first, then by the repository root fallback used by checked-in examples.

Generated finetune, hparam run, infer, final-test, and sleep2stat commands should propagate explicit supported runtime/input fields from the recipe or user decisions. Do not drop explicit checkpoint, pretrained-backbone, eval split, test-after-fit, device, batch-size, precision, split, dry-run, limit-records, survival sidecar, or scheduler/checkpoint cadence fields when rendering scripts.

Hyper-parameter search uses `runtime.<name>` keys for supported CLI knobs and `yaml:/json/pointer/path` keys for generated config overrides. Those exact keys are persisted unchanged in plan, run JSON, workspace, ranking, and adaptive artifacts; removed `param.*` columns are rejected rather than translated. Bare search keys are invalid, and `search.max_runs` must be a positive integer. Hparam recipes inherit base finetune task gates without requiring the base recipe to own a second experiment workspace. Each generated run has a stable `run-NNN` id plus a parameter-derived semantic name, and its config, launch script, metadata, and frozen `runtime_dir`/`checkpoint_dir` links live together under `runs/run-NNN--<semantic-name>/`. `plan.json` stores these entries under `runs` with explicit `experiment_id`, `step_id`, `run_id`, `run_name`, `version`, and snapshot hashes. The generated script working directory is the sole base for relative trainer artifacts: hparam uses an explicit absolute `execution.workdir` or the repository root, while single finetune scripts enforce the repository root. Planned output files must be independent regular files; symlink or hard-link aliases fail before overwrite authorization can mutate another artifact. Existing workspace run-matrix and event files are allowed as derived/append-only outputs but are checked for aliases before a successful plan writes them.

Optional hparam `execution` fields enable active orchestration after a plan has been generated. Every hparam operation crosses the same plan preflight, which verifies workspace ownership, canonical metadata, actual frozen config/script hashes, and exact equality between the runtime-consumed plan recipe (including its resolved base recipe) and `recipe.resolved.yaml` before mutation. Launch, monitor, and stop also preflight every canonical, local-mirror, report, and event target before process, PID, signal, or persistence side effects. Launch additionally checks every exact `stdout.log` and PID target plus its managed directory ancestry before local mkdir/start; SSH execute performs the same check on the execution host, while SSH dry-run makes no remote write or probe. `hparam-launch` rejects completed experiments and validates current tables before creating any run directory or starting a process. For W&B-backed finetune variants, `execution.wandb_project`, `execution.wandb_group`, and `runtime.wandb_mode` are frozen as explicit `--wandb-project`, `--wandb-group`, and `--wandb-mode` arguments in generated finetune scripts; none is a launch-time environment override, and their old `WANDB_*` aliases are rejected in `execution.env`. `execution.gpus_per_run` requires a non-empty physical pool from `execution.gpu_pool` or `runtime.devices`; consultation and runtime both reject an unmanaged implicit pool. The launcher assigns GPUs from the effective pool according to `execution.gpus_per_run` and the real canonical active load, wraps commands with optional `conda run`, nohup logs, and PID files, then writes `launch_manifest.tsv` and `run_status.tsv`. Without explicit `execution.max_concurrent`, each GPU group admits one run; an explicit higher limit permits user-authorized oversubscription and emits a warning. Dry-run is the default; `--execute` is required to start processes, and `run_launched` is emitted only for a process successfully started by that invocation. Adaptive replacement uses the same capacity contract. `execution.path_context=remote` with `path_validation=defer|ssh` lets doctor/plan avoid local false negatives for remote absolute paths; hparam config generation still requires local YAML content when config copies must be written. Logs and PID files are co-located in each managed run directory; deprecated `execution.log_dir` and `execution.pid_dir` fields fail consultation instead of being ignored. `run_manifest.tsv` is the durable status owner, `run_status.tsv` is a validated local mirror that never contributes state, and `launch_manifest.tsv` contributes non-status execution evidence. The workspace owner is the only canonical reader and writer: fresh initialization creates a valid header-only table, every later local/SSH read rejects missing, blank, malformed, legacy, dangling-symlink, duplicate, or recursive state, and every new managed key must match the workspace experiment id. Authoritative experiment/step YAML rejects duplicate keys and recursive aliases, while managed auxiliary tables require current identity headers even when empty. It returns the exact committed rows and passes them directly to mirrors, matrices, reports, and status events. Mutation-facing tracking emits source-allowlisted observations only after workspace plus managed-key ownership is proven and any supplied frozen fields match canonical state. The shared reducer blocks dangerous active-to-scheduled regressions and accepts adaptive `superseded` only when the canonical state re-read by the owner remains `planned/pending`. Canonical execution identity is immutable after its first trusted commit; launch/status evidence cannot initialize it, `hparam-stop` rejects launch drift before PID observation, uses only canonical target/host/PID, fails before PID access when canonical state is terminal, requires `--reason`, and commits `stopped` only after the local signal or SSH command succeeds. PID reads use the same explicit-missing rule as canonical reads: empty/non-numeric/non-positive/invalid-encoding local PID content maps launchable `planned/pending` to non-launchable `missing_pid`; a local path/read `OSError` aborts a planned/pending observation without changing canonical or local state, so a later confirmed absence can launch normally. Other prior states follow the normal reducer, and stop propagates all PID-read failures before kill or mutation. Remote permission/type/transport/timeout failures become `unknown_remote`. `hparam-monitor` validates the same boundary, starts from canonical state, ignores the local mirror, merges only non-status launch evidence, applies one observation, commits canonical first, and mirrors the committed row locally; missing or stale launch status cannot synthesize a regression, and monitoring never launches pending runs. Confirmed remote PID absence keeps active runs non-terminal, uncertain PID reads or process probes become `unknown_remote`, and stop propagates PID-read uncertainty before kill or mutation. Runtime `run_manifest.json` is optional only when truly missing; aliases, non-regular files, invalid encoding/JSON, and non-mapping payloads fail closed in local and mocked SSH paths. `--health` adds opt-in short-command GPU, `/proc/<pid>/io`, log-age, checkpoint-count, and progress-file checks. All hparam postprocessing writers canonicalize the local plan path, preflight the same managed plan, validate complete external candidate tables against the workspace canonical manifest, and require every registered same-step owner plan to share the caller's frozen selection metric and mode before filtering ownership-valid rows from other steps or earlier plans in the same step. Existing checkpoint rankings receive the same ownership/frozen-field validation before runtime scanning. Retained rows additionally prove registered same-step plan ownership and any supplied experiment/version/config/parameter/hash/frozen locator must match before plan fields are overlaid; rank/top-k filtering uses the retained same-step rows in numeric rank order and must retain a candidate before output directories are created. Dry-run replay scripts freeze absolute plan/candidate locators. Removed-format, incomplete, foreign, unmatched, extra, or drifting values fail before output. `hparam-select` preflights ranking output topology before the first ranking or runtime-evidence read, requires all registered hparam plans in a step to use the same validation metric and direction, rebuilds the complete registered step in `reports/ranking.csv` from finite scores, rejects invalid-score rows from other steps instead of silently removing them, and records fixed `epoch=XX.ckpt` paths rather than moving best aliases.

Experiment monitoring commands provide a unified layer above hparam-specific tooling without changing training entrypoints. Local roots are expanded, have `.` and `..` collapsed lexically, and are resolved once before persistence or comparison, except that a direct or dangling symlink at the normalized root itself is rejected before resolution; symlinked parent components may still resolve normally. Recipe-relative roots use the repository root as their base, CLI-relative roots use the current working directory, and both store the same canonical absolute representation. SSH roots remain exact remote strings. `experiment-init` alone accepts a genuinely empty target and creates the canonical empty run table; existing roots require duplicate-key-free id/title/objective/root/baseline equality, a unique rectangular ownership manifest, and strict current-table reads before any write. Every existing `experiment.yaml` is preflighted as a canonical independent regular file before it can establish workspace ownership, in both local and SSH transports. Every other experiment mutation requires initialized, root-matching ownership first. Managed output targets and their workspace subdirectories pass one local/SSH transport preflight that rejects directories, symlinks, hard links, duplicates, and aliases before mutation. Local and SSH modes otherwise follow the same mutation order, and uncertain remote reads fail closed. `experiment-register-step` and the planner read existing steps through the shared owner and merge the canonical `{step, experiment_id, recipe_path, plans}` envelope; only a confirmed missing step may be created, while blank/null/duplicate-key/malformed envelopes and invalid phases fail without repair. Completed experiments reject new plans, and finalization refuses active runs or an empty final report; remote final-report paths must be absolute and are rejected before workspace or SSH access otherwise. `experiment-wandb-sync` uses `wandb.Api()` summary and history only. W&B rows carrying `experiment_id` must match the workspace before managed identity is considered; rows without it ignore unproven step/run claims and may match only one unique runtime version. Foreign and unmatched W&B data stays in raw inventory/history and never enters canonical or managed metrics. `experiment-index-checkpoints` follows only managed runs with a complete frozen `runtime_dir`/`checkpoint_dir` pair; both paths may be empty for non-checkpoint-producing runs, but partial pairs and prior checkpoint rows outside the eligible keys fail before scanning or replacement. Every non-empty checkpoint evidence path must be a direct child of the matched run's frozen `checkpoint_dir`; empty checkpoint evidence remains legal. Workspace metrics/checkpoints prove scope through the validated root plus managed key, and any explicit ownership/frozen drift still fails. It does not recursively scan workspaces or infer identity from directory names. SSH timeout, nonzero exit, malformed row, non-finite mtime, or undeclared checkpoint path leaves the prior checkpoint table untouched. `experiment-monitor` applies the same ownership proof to launch/status evidence, submits current observations through `agent_tools.experiment_workspace`, and builds its payload/report from committed rows. `experiment-rank` preserves `(step_id, run_id)` in `reports/experiment_ranking.csv` without overwriting step-scoped hparam selection. Every managed table boundary states `one_per_run` or `many_per_run`; header-only managed tables retain both identity fields, while removed or incomplete headers fail. Metrics, checkpoint manifests, checkpoint rankings, and selected checkpoint evidence are the multi-row cases. Removed-format workspaces remain available only for manual read-only inspection and are rejected before mutation.

`collect-runs` requires an explicit managed workspace root and is a strict consumer of its authoritative `run_manifest.tsv`: missing, unreadable, or malformed canonical input fails before writing the inventory, a valid header-only table produces an empty inventory, and the requested output may not resolve to the canonical table or use a directory, symlink, hard link, or aliased ancestor. Output topology is validated before parent-directory creation or file opening. Best-effort reading remains limited to optional W&B summaries.

Optional hparam `adaptive` fields enable append-only external-optimized tuning with `adaptive.max_runs_total`. All public local entry paths are canonicalized before use. Initialization completes the shared plan and output-topology preflight before workspace, round, or `adaptive/run_registry.tsv` writes; `workflow.json` stores absolute workflow/recipe locations, registry rows copy the frozen plan's absolute config/script paths and round directory, and generated round recipes resolve their experiment root and base recipe. Later digest, suggestion, step, and loop operations reject relative workflow recipe locators, validate every registry row against the canonical workspace, and preflight deterministic digest/incumbent/suggestion/registry/event outputs before monitoring or writing. Preview steps monitor current status and write digest, suggestion, and preview events, but do not stop or supersede runs or create and launch another round. Execute mode may create and register a prospective replacement round only when it fits both round and total-run budgets. Prospective round numbers advance above every registered row and existing `round_NNN` directory, while the round budget counts only committed rounds; therefore a failed directory is never overwritten or resumed, and a later step can use a fresh round number without rewriting prior artifacts. A failed plan registration or zero-start launch supersedes that uncommitted plan's canonical `planned/pending` rows before returning. Execute first launches against real free capacity; when replacements remain pending in a full pool, it stops at most one evidence-qualified bad running row before an ordinary launch retry and requires a newly confirmed replacement before any further retirement. Apart from the single row drained for the current launch attempt, confirmed replacements and retired bad rows remain one-to-one. The first replacement confirmed `launched` or `running` writes the existing `launch_round` commit event even if a sibling records `launch_failed`, after which old pending runs may be superseded. Launcher exceptions report stopped current runs separately from superseded current pending runs, and an existing commit is never rolled back. A pre-drain failure with no canonically confirmed replacement leaves old runs unchanged and the prior round current. A zero-start failure after draining preserves the recorded stop, terminalizes remaining prospective pending rows, and leaves the prior round current; a later failure after commit leaves the new round current and remaining old runs unchanged. Adaptive lifecycle events, digest paths, suggestion paths, and round locators are absolute; earlier plans, configs, logs, and checkpoints are not rewritten.

Preset generation commands should render explicit supported `preset:` fields such as stride, channels, metadata, missing-channel policy, output template, overwrite, dry-run, manifest, and sidecar-manifest flags.

## Reuse Guidance

Do not create a second trainer, preprocessing runtime, or sleep2stat execution manager. Generated commands must call the existing variant package entrypoints, `preprocess/save_dataset_presets.py`, `preprocess/convert_npz_to_kaldi.py`, `sleep2vec2/preprocess/*`, `sleep2expert/preprocess/*`, `python -m sleep2stat`, and `utils/check_configs.py`. Long-running preprocessing should keep tqdm for humans and write `status/progress.json` for agents; `agent_tools progress` reads that file locally or through a single short SSH `cat`, and invalid or half-written progress JSON should report `status=unknown` rather than crash. Current progress writers cover root and variant Kaldi conversion, preset generation, preset merging, index splitting, missing-mask statistics, WatchPAT EDF conversion, `utils/fill_index_duration.py`, `utils/cut_ukb_sleep_with_asleep.py`, `utils/collect_ukb_demographics.py`, and `utils/parse_ukb_annotations_by_person.py`.

## Edit Hotspots

- Change consultation policy/source resolution: `agent_policies/consultation_policy.yaml` plus `agent_tools/decisions.py`; change path, task, or tuning rules in `decision_paths.py`, `decision_rules.py`, or `decision_hparam.py`.
- Change recipe loading or base-recipe inheritance: `agent_tools/recipes.py`.
- Change plan orchestration/task dispatch: `agent_tools/plans.py`; change CLI rendering in `plan_rendering.py`, context/index/Markdown in `plan_context.py`, or grid/final-test/frozen-plan behavior in `plan_hparam.py`.
- Change skill validation: `agent_tools/skills.py`.
- Change hparam launch/monitor/stop in `hparam_runtime.py`, selection/checkpoint scans in `hparam_selection.py`, or external evaluation/logit/threshold/ensemble behavior in `hparam_postprocess.py`; keep `agent_tools/hparam.py` as the public facade.
- Change experiment lifecycle orchestration in `agent_tools/experiments.py`, local/SSH persistence in `experiment_io.py`, or W&B/checkpoint/run monitoring/ranking in `experiment_tracking.py`.
- Change adaptive hparam workflow: `agent_tools/adaptive_hparam.py`.
- Change progress artifacts: `agent_tools/progress.py`.
